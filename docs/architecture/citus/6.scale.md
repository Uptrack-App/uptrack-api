

HA postgres:
Minimum for PostgreSQL HA: 3 nodes

  | Node Count | Setup                                                | Verdict
                            |
  |------------|------------------------------------------------------|------------------
  --------------------------|
  | 2 nodes    | Primary + Standby                                    | ❌ No automatic
  failover (split-brain risk) |
  | 3 nodes    | Primary + 2 Standby (or Primary + Standby + Witness) | ✅ Minimum for
  quorum                       |
  | 5 nodes    | Primary + 4 Standby                                  | For higher fault
  tolerance                 |

  Why 3?

  Consensus-based failover requires a majority vote (quorum):
  - 2 nodes: Need 2/2 agreement → 1 failure = no quorum
  - 3 nodes: Need 2/3 agreement → 1 failure = still have quorum
  - 5 nodes: Need 3/5 agreement → 2 failures = still have quorum

  Common 3-node patterns:

  Option A: 3 full Postgres nodes
  ┌─────────┐  ┌─────────┐  ┌─────────┐
  │ Primary │  │ Standby │  │ Standby │
  │  (sync) │──│ (sync)  │──│ (async) │
  └─────────┘  └─────────┘  └─────────┘

  Option B: 2 Postgres + 1 witness (cheaper)
  ┌─────────┐  ┌─────────┐  ┌─────────┐
  │ Primary │  │ Standby │  │ Witness │
  │  (data) │──│ (data)  │──│(no data)│
  └─────────┘  └─────────┘  └─────────┘

  Tools for Postgres HA:
  - Patroni - Most popular, etcd/consul-based
  - Stolon - Kubernetes-native
  - pg_auto_failover - Simpler, Postgres-native
  - Citus - Distributed/sharded

  Recommendation for your project: Start with 3 nodes using Patroni - it's battle-tested
  and works well with NixOS.

  | Scaling Type                  | Write Throughput       | How
        |
  |-------------------------------|------------------------|-----------------------------
  ------|
  | Read replicas (Patroni alone) | ❌ No improvement       | All writes go to single
  primary   |
  | Sharding (Citus)              | ✅ Yes, scales linearly | Writes distributed across
  workers |

  Why read replicas don't help writes:

  All writes → Single Primary → Replicated to standbys
                     ↓
              Bottleneck here

  Why sharding helps writes:

  Writes split by shard key (e.g., tenant_id)
           │
      ┌────┼────┬────┐
      ▼    ▼    ▼    ▼
   Worker1 Worker2 Worker3 Worker4
   (25%)   (25%)   (25%)   (25%)

  4 workers = ~4x write throughput

  Citus write scaling example:

  -- Shard by tenant_id
  SELECT create_distributed_table('events', 'tenant_id');

  -- This INSERT routes to ONE worker (fast)
  INSERT INTO events (tenant_id, data) VALUES (123, '...');

  -- Writes for different tenants go to different workers (parallel)

  Caveats:

  | Limitation                | Impact                           |
  |---------------------------|----------------------------------|
  | Cross-shard transactions  | Slower (2PC coordination)        |
  | Cross-shard JOINs         | Slower (data shuffle)            |
  | Bad shard key choice      | Hot spots, uneven distribution   |
  | Single-tenant heavy write | Still bottlenecked on one worker |

  Bottom line:

  - Need more read throughput → Add read replicas (Patroni)
  - Need more write throughput → Shard the data (Citus)
  - Need both → Citus + Patroni on each worker
