## ADDED Requirements

### Requirement: All UI strings are externalized through Paraglide
The frontend SHALL use Paraglide JS for all user-visible strings. No hardcoded English string literals SHALL appear in component JSX or TypeScript outside of `messages/en.json`. Strings SHALL be accessed via the typed `m.*()` functions generated by the Paraglide Vite plugin at build time.

#### Scenario: String accessed via typed function
- **WHEN** a component renders a UI string
- **THEN** it calls `m.string_key()` from the generated `@/paraglide/messages` module
- **THEN** the build fails if the key does not exist in `messages/en.json`

#### Scenario: English fallback for missing translations
- **WHEN** a message key exists in `messages/en.json` but not in `messages/ja.json`
- **THEN** the Japanese locale renders the English string (no empty/missing text)

---

### Requirement: Paraglide locale synced with user preference
The active Paraglide locale SHALL be initialized from the user's `preferred_locale` on app load and updated immediately when the user changes their language. The `<html lang>` attribute SHALL reflect the active locale.

#### Scenario: Locale initialized from user record
- **WHEN** the app loads for an authenticated user with `preferred_locale = 'ja'`
- **THEN** Paraglide's active locale is `'ja'`
- **THEN** `<html lang="ja">` is set on the root element

#### Scenario: Locale updates without page reload
- **WHEN** the user changes their language to German in Settings
- **THEN** `setLocale('de')` is called
- **THEN** all `m.*()` strings immediately render in German without a full page reload

---

### Requirement: Dates and times formatted using Intl API
The frontend SHALL format all displayed dates, times, and durations using `Intl.DateTimeFormat` with the active locale. No hardcoded date format strings (e.g., `MM/DD/YYYY`) SHALL be used.

#### Scenario: Date formatted for Japanese locale
- **WHEN** an incident timestamp is displayed with locale `ja`
- **THEN** the date renders as `2026年4月14日` (or equivalent locale-appropriate format)

#### Scenario: Date formatted for German locale
- **WHEN** an incident timestamp is displayed with locale `de`
- **THEN** the date renders as `14. April 2026` (or equivalent locale-appropriate format)

---

### Requirement: Numbers and percentages formatted using Intl API
The frontend SHALL format uptime percentages, response times, and other numeric values using `Intl.NumberFormat` with the active locale. Decimal and thousands separators SHALL adapt to locale conventions.

#### Scenario: Uptime percentage in German locale
- **WHEN** uptime is displayed as `99.95%` with locale `de`
- **THEN** it renders as `99,95 %` (comma decimal separator per German convention)

#### Scenario: Response time in English locale
- **WHEN** response time `1234` ms is displayed with locale `en`
- **THEN** it renders as `1,234 ms`

---

### Requirement: Message files maintained for all supported locales
The frontend SHALL maintain translation JSON files for each supported locale at `messages/{locale}.json`. Each file SHALL contain the same set of message keys as `messages/en.json`. Missing keys fall back to English (see Paraglide fallback config).

#### Scenario: All locale files present at build time
- **WHEN** the Vite build runs
- **THEN** `messages/en.json`, `messages/ja.json`, `messages/de.json`, `messages/es.json`, and `messages/pt-BR.json` all exist
- **THEN** the build succeeds

#### Scenario: New string added to English file
- **WHEN** a developer adds a new key to `messages/en.json`
- **THEN** the TypeScript type for `m.*()` is updated automatically by the Paraglide plugin
- **THEN** components using the key compile successfully
- **THEN** other locale files without the key fall back to the English value at runtime
